# Task 5.1：API 总体规范与安全规则设计

## 1. 文档目的

本文档统一 Violin ERP Lite 的 API 总体规范、安全规则和通用接口契约，作为后续 Task 5.2 至 Task 5.5 的共同设计基线。

本文件用于统一：

- PC 管理端与微信小程序的接口规范；
- 后续业务接口文档的描述格式；
- 身份认证、权限校验、数据权限、状态校验和输入校验原则；
- 幂等、并发、错误处理、审计日志和数据脱敏原则；
- 两端共用同一套后端接口、数据和 Frozen 业务规则的约束。

本任务只完成接口设计规则，不创建真实 API Route，不编写业务代码，不选择技术框架，不修改数据库或页面。


如 API 需求与上游文件冲突，必须停止冲突部分的设计，记录文件、位置和影响，不得自行推翻、绕过或变相扩展上游设计。

## 3. API 设计原则

1. API 采用表述性状态转移（RESTful）风格和资源导向设计。
2. 资源名称表达正式业务资源，不以页面名称、按钮名称或前端组件名称代替业务资源。
3. PC 管理端与微信小程序共用业务接口、数据库和业务规则；不同终端只能在登录入口、交互节奏和展示能力上存在差异。
4. 已发布接口契约必须保持稳定；不兼容变更必须按版本规则处理。
5. 普通创建或更新接口不得直接修改 `inventories` 库存余额，也不得手工生成无合法来源的 `inventory_transactions`。
6. 状态变更必须使用专用动作接口；客户端不得通过普通 `PUT` 或 `PATCH` 直接覆盖状态字段。
7. 提交、审核、反审核、取消、作废、冲销和正式库存动作不得使用普通更新接口模拟。
8. 后端必须重新执行身份认证、权限、数据范围、当前状态、业务前置条件和输入数据校验。
9. 客户端传入的用户、角色、权限、汇总数量、库存数量、金额汇总和目标状态均不得直接作为可信事实。
10. 业务资源、字段、状态和关联关系必须映射现有 Approved 或 Frozen 对象；接口层不得建立平行业务事实来源。
11. 查询汇总只用于展示，不得形成独立于正式业务表和库存表的第二套数据源。
12. 所有关键写操作必须可追踪、可审计，并满足历史记录保护规则。

## 4. API 版本规则

### 4.1 基础路径

第一版接口统一使用：

```text
/api/v1
```

版本信息在 URL 中表达，URL 中的主版本是接口版本唯一依据；本阶段不要求重复发送自定义版本 Header。

### 4.2 主版本升级条件

以下不兼容变化必须升级主版本：

- 删除已发布字段；
- 修改字段类型、含义、单位、精度或空值语义；
- 将可选字段改为必填字段；
- 修改资源路径或 HTTP 方法；
- 修改成功或失败响应的基础结构；
- 修改已发布错误码含义；
- 修改分页、排序或筛选参数的既有语义；
- 其他导致既有客户端无法继续按原契约工作的变化。

### 4.3 向后兼容与字段演进

- 同一主版本内优先保持向后兼容；
- 新增响应字段必须为客户端可忽略的可选字段；
- 新增请求字段默认应为可选字段，除非进入新主版本；
- 不得无提示删除字段、改名或改变字段含义；
- 字段废弃必须先在接口文档中标记，说明替代字段和计划停止支持的版本；
- 废弃期内旧字段语义必须保持稳定；
- API 版本升级不得借机改变 Frozen 业务语义。

本阶段只设计版本规则，不创建任何 Route。

## 5. URL 与资源命名规范

### 5.1 命名规则

- URL 只使用英文小写字母、数字和必要的路径分隔符；
- 资源名称使用复数名词；
- 多个单词使用短横线连接；
- 禁止使用中文、驼峰命名、下划线或页面名称；
- 资源标识使用路径参数 `{id}`；
- 查询、筛选、分页和排序使用 Query 参数；
- 动作路径使用清晰、稳定的英文动词。

### 5.2 规范示例

以下路径仅用于说明规范，不代表已创建真实接口：

```text
GET  /api/v1/products
GET  /api/v1/products/{id}
POST /api/v1/purchase-orders
POST /api/v1/purchase-orders/{id}/submit
POST /api/v1/purchase-orders/{id}/approve
POST /api/v1/purchase-orders/{id}/cancel
```

### 5.3 更新与状态动作

- `PUT` 用于上游规则允许的整体替换，客户端必须提交完整可更新表示；缺失字段不得被静默解释为删除。
- `PATCH` 用于上游规则允许的局部更新；未出现字段表示不修改，显式 `null` 只允许清空可空且允许清空的字段。
- 正式单据状态、审核状态和库存结果不得通过 `PUT` 或 `PATCH` 直接修改。
- 提交、审核、反审核、取消、作废、冲销和库存执行使用专用动作路径，并再次校验状态与权限。

### 5.4 删除、停用、作废与取消

- 删除表示物理移除，仅可在 Approved/Frozen 规则明确允许且不破坏历史时存在；正式业务记录默认禁止物理删除。
- 停用适用于可停用的基础资料，保留历史引用，并通过专用动作表达。
- 作废适用于已不再有效但必须保留历史的正式单据或记录。
- 取消适用于尚未形成不可逆业务结果、且状态流转允许终止的业务。
- 上述动作含义不得互相替代，也不得统一包装为普通 `DELETE`。

## 6. HTTP 方法与状态码规范

### 6.1 HTTP 方法

| 方法 | 使用边界 |
| --- | --- |
| `GET` | 查询资源或集合，不得产生业务状态变化 |
| `POST` | 创建资源或执行提交、审核、取消等专用业务动作 |
| `PUT` | 对允许整体更新的资源提交完整可更新表示 |
| `PATCH` | 对允许局部更新的资源更新指定字段，不得直接覆盖状态 |
| `DELETE` | 仅用于上游规则明确允许物理删除的非历史资源；正式业务资源默认禁用 |

### 6.2 成功状态码

| 状态码 | 使用规则 |
| --- | --- |
| `200 OK` | 查询、更新或同步动作成功并返回响应体 |
| `201 Created` | 资源创建成功；响应返回新资源或其标识 |
| `202 Accepted` | 请求已受理并进入异步任务，尚未完成最终业务结果 |
| `204 No Content` | 操作成功且无需响应体；使用时不返回统一响应包装 |

### 6.3 客户端与业务错误状态码

| 状态码 | 使用规则 |
| --- | --- |
| `400 Bad Request` | JSON、Header、Query 或路径格式无法解析 |
| `401 Unauthorized` | 未提供有效登录凭证、登录已失效或凭证无效 |
| `403 Forbidden` | 已认证但缺少功能、操作、数据范围或记录级权限 |
| `404 Not Found` | 资源不存在，或按安全策略对无权获知存在性的资源返回不可见 |
| `409 Conflict` | 并发版本、状态竞争、幂等键复用、唯一性竞争或库存并发冲突 |
| `422 Unprocessable Entity` | 请求格式正确，但字段或业务规则校验未通过 |
| `429 Too Many Requests` | 超出访问频率或安全限制 |

### 6.4 服务端错误状态码

| 状态码 | 使用规则 |
| --- | --- |
| `500 Internal Server Error` | 未预期的服务端错误；不得暴露内部异常详情 |
| `503 Service Unavailable` | 依赖或服务暂时不可用，可结合安全的重试提示 |

业务错误不得只依赖 HTTP 状态码，必须同时返回稳定的项目错误码。HTTP 状态码表达错误类别，项目错误码表达具体业务原因。

## 7. 统一请求规范

### 7.1 内容与 Header

- JSON 请求使用 `Content-Type: application/json; charset=utf-8`；
- 客户端声明 `Accept: application/json`；
- 认证凭证通过标准 `Authorization` Header 或后续技术阶段批准的安全会话机制传递；
- 禁止在 URL、Query 参数或页面可见地址中传递密码、Token 或其他认证凭证；
- `X-Request-ID` 用于端到端请求追踪；客户端可提供合法值，缺失或不合法时由服务端生成；
- `Idempotency-Key` 用于需要幂等保护的写操作；
- `X-Client-Type` 标识 `pc` 或 `wechat-mini-program`，仅用于兼容、审计和监控，不作为授权依据；
- API 主版本由 URL 表达，不使用客户端 Header 覆盖 URL 版本。

Header 名称不区分大小写，但文档统一使用上述写法。服务端不得信任客户端伪造的用户、角色、IP、设备或终端来源信息。

### 7.2 JSON 字段命名

- 通用请求与响应字段使用英文 `lowerCamelCase`；
- URL 资源名称使用英文小写短横线形式；
- 业务字段必须明确映射 Frozen 字段语义，不得因命名转换产生新的业务含义；
- 后续 Task 必须在接口模板中列出数据库对象来源和字段映射依据。

### 7.3 数据格式

| 数据类型 | API 格式 |
| --- | --- |
| ID | UUID 字符串；依据 Frozen 设计并优先由服务端生成 UUID v7 |
| 日期 | `YYYY-MM-DD` |
| 日期时间 | ISO 8601 字符串，必须包含时区或 `Z` 标识 |
| 金额、数量、单价、比例 | 十进制定点字符串，精度遵循 Frozen `DECIMAL` 规范，禁止浮点误差改变业务值 |
| 布尔值 | JSON `true` 或 `false`，不得使用 `0/1`、`yes/no` |
| 枚举 | 已批准的稳定小写英文代码，不得由客户端提交显示文案代替 |
| 空值 | 使用 JSON `null`；仅用于正式定义为可空且允许清空的字段 |

创建请求中的必填字段不得缺失或为 `null`。局部更新中，字段缺失表示不修改；显式 `null` 与字段缺失不得混为一谈。

接口层不得重新定义 Frozen 数据库中不存在的业务字段、表、状态、约束或关联关系。展示型计算值必须明确标注为实时派生数据，并说明正式数据来源。

## 8. 统一响应结构

### 8.1 成功响应

除 `204 No Content` 外，成功响应统一为：

```json
{
  "success": true,
  "data": {},
  "meta": {},
  "requestId": "string"
}
```

- `success`：表示本次 HTTP 请求是否成功完成；
- `data`：承载单个资源、资源集合、动作结果或异步任务受理结果；无业务数据时可为 `null`；
- `meta`：承载分页、排序、筛选摘要、数据日期或其他非业务主体元数据；不适用时返回空对象；
- `requestId`：服务端确认的请求追踪标识，必须与日志中的 `request_trace_id` 对应。

### 8.2 失败响应

失败响应统一为：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_INVALID_FIELD",
    "message": "请求数据校验失败",
    "details": []
  },
  "requestId": "string"
}
```

- `error.code`：稳定的项目错误码；
- `error.message`：面向用户或业务人员的中文错误说明；
- `error.details`：可选的字段级或明细行级错误数组，不得包含密码、Token、完整敏感值或内部实现信息；
- `requestId`：用于用户报障、日志查询和跨服务追踪。

任何响应均不得向客户端返回异常堆栈、SQL、数据库结构、服务器文件路径、内部类名、密钥、配置值或未脱敏的内部异常详情。

## 9. 分页、排序、筛选与搜索规则

### 9.1 分页

- `page`：页码，从 `1` 开始，默认值为 `1`；
- `pageSize`：每页记录数，默认值为 `20`，最大值为 `100`；
- 超过最大值时返回 `422`，不得静默扩大查询；
- 列表响应的 `meta` 至少返回 `page`、`pageSize`、`total` 和 `totalPages`；
- `total` 和 `totalPages` 只统计当前用户有权访问的数据。

### 9.2 排序

- `sortBy` 指定排序字段；
- `sortOrder` 只允许 `asc` 或 `desc`；
- 每个接口必须定义可排序字段白名单；
- 禁止将客户端字段名直接拼接为数据库排序表达式；
- 未指定排序时，后续业务接口必须定义稳定默认排序，并使用唯一标识作为必要的次级排序依据。

### 9.3 筛选与搜索

- 精确筛选使用对应字段参数；
- 范围筛选使用明确的 `From`、`To`、`Min` 或 `Max` 后缀；
- 日期范围必须明确是否包含边界，默认开始和结束日期均包含；
- 状态筛选只能使用上游已批准状态值；
- 关键词搜索统一使用 `keyword`，每个接口必须说明搜索字段范围；
- 仓库和店铺筛选分别使用 `warehouseId` 和 `storeId`，并执行服务端数据权限校验；
- 空字符串不得与 `null` 或未提供参数混为一谈；
- 无权限数据必须在服务端查询或聚合前过滤；
- 客户端不得通过传入仓库、店铺、角色或其他范围参数扩大自身数据权限。

## 10. 身份认证规则

1. 所有非公开接口必须验证有效登录态。
2. Token、会话、签名和 Cookie 的具体技术方案留待后续技术阶段确定，本任务不选择认证框架。
3. 未认证、凭证无效或登录失效统一返回 `401` 和 `AUTH_*` 错误码。
4. 客户端收到登录失效响应后，应清理无效登录态并引导重新登录；不得自动无限重试写操作。
5. 禁止在 URL、日志、错误信息和页面可复制地址中传递或输出认证凭证。
6. 密码只能依据现有 `users.password_hash` 语义处理，不得保存、返回或记录明文密码。
7. PC 管理端和微信小程序可以使用不同登录入口，但认证结果必须统一映射到现有 `users`、`user_roles`、`roles`、`role_permissions` 和 `permissions` 体系。
8. 本任务不新增用户表、认证表、Token 表或认证字段。

## 11. 权限校验规则

后端必须组合校验以下权限维度：

- 功能权限：是否可以访问模块或资源；
- 操作权限：是否允许查看、新增、编辑、提交、审核、导出等具体动作；
- 数据权限：是否能访问对应仓库、店铺及上游规则限定的数据范围；
- 状态权限：当前业务状态是否允许该角色执行该动作；
- 角色权限：用户当前有效角色是否授予对应权限；
- 仓库权限：依据 `role_warehouses` 及其 `access_level`；
- 店铺权限：依据 `role_stores` 及其 `access_level`；
- 记录级权限：依据现有业务关联、创建人、负责人、来源仓库、店铺或厂家仓等正式关系推导，不建立通用记录 ACL；
- 审核分离：制单、审核、仓库执行和关键调整遵循 Frozen 权限分离原则。

客户端按钮隐藏、路由守卫或页面只读状态仅用于交互提示，不能替代后端校验。每次读取和写入均必须在服务端重新计算有效权限，不得信任客户端传入的角色或权限码。

`401` 表示未通过身份认证；`403` 表示身份已确认但权限不足。为防止敏感资源枚举，具体接口可按安全规则对无权获知存在性的资源返回 `404`，但内部日志必须保留真实拒绝原因。

## 12. 仓库与店铺数据权限

1. 用户只能查询其有效角色授权的仓库和店铺数据。
2. 创建单据时，必须校验来源仓、目标仓、在途仓、店铺及其他适用范围的操作权限。
3. 修改、提交、审核、取消、作废及正式执行时必须重新校验，不得沿用页面打开时的旧权限结果。
4. 列表总数、Dashboard 指标、统计汇总和导出只能包含授权范围内的数据。
5. 跨仓库或跨店铺操作必须同时满足全部相关端点的权限要求；任一端无权即拒绝整个动作。
6. 厂家及厂家仓范围必须通过现有厂家仓和仓库授权关系表达，不新增平行厂家权限字段。
7. 禁止通过修改 `warehouseId`、`storeId`、来源单据 ID 或路径参数访问其他范围的数据。
8. 超级管理员或系统角色的高风险操作同样必须记录审计日志，不得成为无痕操作。
9. 本任务不新增权限表、权限字段或用户直接权限关系。

## 13. 状态校验规则

1. 所有状态操作必须在服务端依据 Approved 状态流转设计校验。
2. 客户端不得提交任意目标状态并覆盖当前状态。
3. 专用动作必须同时校验当前状态、允许的目标状态、业务前置条件、操作权限和数据权限。
4. 非法状态跳转返回 `409` 或 `422`，并使用稳定的 `STATE_*` 错误码说明原因。
5. 重复提交同一动作时，先执行幂等校验；不得重复生成审批记录、库存流水或下游单据。
6. 已完成记录原则上只读，纠错必须使用上游批准的冲销、调整或正式纠错流程。
7. 已取消或已作废记录原则上只读，不得重新作为有效业务执行。
8. 审核、反审核、取消、作废和冲销动作只在上游设计明确允许时存在。
9. 反审核只允许在尚未形成库存流水或后续关联单据时执行。
10. 状态变化必须保留 `document_status_histories`、`approval_records` 及必要 `audit_logs` 记录，不得覆盖历史。
11. 不得创造新的状态值，也不得用页面筛选名称替代正式状态。

## 14. 数据校验规则

API 必须执行并返回明确结果的校验至少包括：

- 必填字段校验；
- JSON 类型和业务类型校验；
- 字符串长度校验；
- 数量、金额、单价、比率的精度和范围校验；
- 枚举值白名单校验；
- 编码、编号和其他业务唯一性校验；
- 关联对象存在性校验；
- 关联对象启用状态校验；
- 业务日期和时间先后关系校验；
- 数量、金额及合计关系合法性校验；
- 正式单据明细非空校验；
- 主从归属和来源明细一致性校验；
- 跨字段条件必填、互斥和合计校验；
- Frozen Check Constraint 对应的同表逻辑校验；
- 权限、状态、库存及跨表业务规则校验。

接口校验用于尽早返回可理解的业务错误，但不能取代数据库主键、唯一约束、外键、Check Constraint 或事务一致性保护。并发情况下仍必须处理数据库约束竞争结果。

## 15. 幂等规则

### 15.1 适用操作

以下写操作原则上必须支持幂等保护：

- 创建正式业务单据；
- 提交、审核、反审核、取消和作废；
- 正式入库、出库、调拨、库存调整、报损及其他库存事务；
- Excel 正式导入和导入执行；
- 可能因网络超时或客户端重试而重复执行的高风险动作。

`GET` 查询不要求 `Idempotency-Key`，但仍必须保持无副作用。

### 15.2 幂等键规则

- 幂等键通过 `Idempotency-Key` Header 传递；
- 幂等范围至少包含认证主体、请求方法、规范化路径和幂等键；
- 同一幂等键和相同请求内容重复提交时，应返回首次已确认结果，不得重复执行；
- 同一幂等键对应不同请求内容时返回 `409` 和 `CONFLICT_*` 错误码；
- 请求体摘要计算、结果保存介质、保留时限和清理策略留待后续技术设计；
- 客户端重试必须复用原幂等键，不得每次生成新键规避保护；
- 幂等成功不得掩盖首次请求的业务失败结果。

本任务不为幂等新增数据库表或字段。后续实现若无法使用获批准的技术机制，且需要修改 Frozen 数据库结构，必须提出 Database Change Request。

## 16. 并发控制规则

1. 提交、审核、反审核和状态动作必须防止重复执行。
2. 正式入库、出库、调拨、调整、报损和跨境库存动作必须防止重复库存流水与重复余额变更。
3. 出库及库存减少操作必须防止并发超扣和负库存。
4. 客户端更新正式业务单据时必须提交其读取到的 `versionNo`；服务端依据 Frozen `version_no` 校验并原子递增。
5. 对不存在 `version_no` 的对象，只能使用其已批准的时间字段、唯一约束、当前状态和数据库事务进行并发保护，不得擅自新增版本字段。
6. 旧页面或旧请求不得静默覆盖新数据；版本或状态不一致返回 `409`。
7. 状态校验、状态写入、状态历史、审批记录和审计记录必须在一致的原子边界内完成。
8. 库存读取、可用量校验、余额写入和库存流水记录必须处于同一数据库事务，并采用后续技术阶段批准的并发控制方式。
9. 事务失败必须整体回滚，不得出现只改余额未写流水、只写流水未改余额或调拨单边完成。
10. 唯一约束竞争、幂等冲突和并发版本冲突统一使用 `409`，并返回可识别的具体错误码。

本任务不选择锁、隔离级别或 ORM 实现。后续若现有字段、约束和事务无法满足一致性要求，必须停止实现并提出 Database Change Request。

## 17. 错误码总体规范

### 17.1 分类

| 分类 | 含义 |
| --- | --- |
| `AUTH_*` | 身份认证、登录态和凭证错误 |
| `PERMISSION_*` | 功能、操作和数据权限错误 |
| `VALIDATION_*` | 格式、必填、类型、精度和跨字段校验错误 |
| `RESOURCE_*` | 资源不存在、停用或不可用 |
| `STATE_*` | 当前状态不允许目标动作或状态已变化 |
| `CONFLICT_*` | 并发、幂等、唯一性和重复操作冲突 |
| `INVENTORY_*` | 库存不足、负库存风险、流水或库存事务错误 |
| `IMPORT_*` | 文件、校验、匹配、重复文件和执行错误 |
| `SYSTEM_*` | 未预期错误或暂时不可用 |

### 17.2 稳定性规则

- 错误码使用大写英文与下划线，含义稳定；
- 错误信息使用中文，并给出可执行的修正提示；
- 同一错误不得在不同接口使用不同含义；
- 不得将数据库异常文本、SQL、服务器路径或堆栈作为错误信息；
- 错误详情只能包含定位业务问题所需的字段、明细行和安全摘要；
- 具体业务错误码由 Task 5.2 至 Task 5.4 补充，Task 5.5 最终汇总；
- 已发布错误码废弃必须遵循版本和兼容规则。

## 18. 操作日志规则

### 18.1 必须记录的操作

- 登录成功、失败、锁定、失效等安全事件；
- 基础资料新增、修改、启用和停用；
- 正式单据新增、修改、提交、审核、反审核、取消和作废；
- 入库、出库、调拨、盘点调整、库存调整、报损和跨境库存动作；
- Excel 上传、校验、确认和正式导入；
- 附件关联、解除关联或删除动作；
- 用户、角色、权限、仓库范围和店铺范围变更；
- 包含敏感信息或高风险数据的导出。

### 18.2 日志内容

日志至少关联：

- 操作人或合法系统事件主体；
- 操作时间；
- 模块与业务对象；
- 业务记录标识和必要编号快照；
- 操作类型；
- `requestId`，对应 `audit_logs.request_trace_id`；
- 操作结果与安全的失败原因；
- 适用变更的必要前后摘要。

业务操作日志统一使用现有 `audit_logs`，状态历史使用 `document_status_histories`，审核与反审核使用 `approval_records`。登录失败等无法映射现有业务记录的安全事件，其持久化映射必须在后续技术设计中验证；不得伪造业务对象标识，也不得擅自新增日志表或字段。如必须修改 Frozen 结构，需提出 Database Change Request。

日志不得保存明文密码、Token、Cookie、完整银行账号、完整地址、完整联系方式、文件原始敏感内容或其他无需留存的敏感参数。

## 19. 数据脱敏规则

1. 密码哈希、认证凭证、Token、Cookie 和安全密钥不得通过业务接口返回。
2. 手机号、邮箱及其他联系方式根据角色和业务必要性部分隐藏。
3. 客户姓名、收件地址及销售出库个人信息默认按最小必要原则展示。
4. 供应商和生产厂家联系人、税务识别号、银行名称、账户名称和银行账号按字段权限展示并脱敏。
5. 付款账户快照、银行参考号和金额信息只向具备对应权限的用户展示。
6. 日志中的请求参数、响应摘要和失败原因必须在写入前脱敏。
7. 导出必须复用查询权限和字段权限，并记录导出范围、操作者和结果。
8. 脱敏只改变展示，不改变正式数据；无权限用户不得通过排序、搜索、错误信息或导出推断原值。
9. 脱敏规则只能覆盖 Frozen 数据库中已存在或 Approved 页面已引用的敏感字段，不得假定存在新的敏感字段。

## 20. API 安全规则

- 生产环境所有接口必须通过 HTTPS 提供，禁止明文传输凭证和业务数据；
- 每次请求执行身份认证、功能权限、数据权限和记录级权限校验；
- 所有 Path、Query、Header、JSON 和文件输入必须校验类型、长度、范围和白名单；
- 响应仅返回当前操作所需字段，并执行字段权限和脱敏；
- 列表、批量查询、批量导出和批量动作逐项防止水平越权和批量越权；
- 高风险写操作执行幂等、状态和并发保护；
- 服务端必须限制请求体、批量记录数、文件大小和执行时长，具体阈值由后续技术阶段确定；
- 文件上传必须校验扩展名、MIME、内容特征、大小和安全性；
- 按接口风险实施访问频率限制，超限返回 `429`；
- CORS 默认拒绝未授权来源，只允许正式配置的 PC 与微信小程序相关来源；
- 如后续采用基于 Cookie 的会话认证，必须实施 CSRF 防护；如采用 Header Token，仍必须防止跨站泄露和错误持久化；
- 数据访问使用参数化查询或 ORM 安全绑定，禁止拼接不可信输入；
- 输出和富文本按场景编码，防止脚本注入；
- 文件路径、对象键和下载参数必须规范化并限制范围，防止路径穿越；
- 不向客户端暴露服务器路径、数据库结构、内部网络信息或异常堆栈；
- 生产环境禁止调试输出和详细异常页面。

具体认证库、网关、限流器、CORS 中间件、CSRF 方案和安全扫描工具留待后续技术阶段确定。

## 21. 附件接口通用原则

1. 文件上传必须关联合法且有权限访问的业务对象，使用现有 `attachments` 和 `attachment_links`。
2. 上传前校验业务对象存在性、对象状态、操作权限和附件类别。
3. 同时校验扩展名、MIME、文件内容特征和文件大小；禁止可执行文件及伪装文件。
4. 原始文件名只用于安全展示，不得直接作为存储路径或可执行文件名。
5. 文件存储只保存正式元数据和存储引用，不将文件本体写入数据库。
6. 下载前必须重新校验当前用户的业务对象权限、字段权限和附件敏感级别。
7. 解除关联、停用或删除附件必须符合历史保护规则并记录审计日志；不得通过物理删除破坏正式业务历史。
8. 下载响应不得暴露服务器路径或内部存储凭证。
9. 实际附件接口目录、文件阈值和允许类型在 Task 5.5 设计，本任务不创建接口。

- 状态；
