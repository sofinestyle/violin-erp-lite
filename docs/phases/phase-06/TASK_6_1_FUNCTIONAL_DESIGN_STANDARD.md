# Task 6.1：功能详细设计统一规范（Functional Design Standard）

## 1. 文档目的与适用范围

本文定义 Phase 6 全系统统一的功能详细设计规范，供 Task 6.2 核心业务功能详细设计和 Task 6.3 公共能力功能详细设计直接引用。通用规则只在本文件定义一次，后续功能文档必须引用本规范并补充具体功能差异。

Phase 6 只定义已批准功能如何运行，不重新定义业务模块、页面、数据库或 API。Frozen 业务规则、Database Logical Design v1.1、API Master Specification v1.1 及 Approved 页面设计的优先级高于本规范。

发现本规范或后续功能需求无法映射 Frozen 内容时，必须停止冲突部分，不得通过页面逻辑、临时字段或实现代码绕过，并登记正式 DCR 或 Change Request 等待项目负责人裁定。

本文件不指定最终技术框架、代码结构、组件库或第三方工具，不创建真实页面、API Route 或业务代码；相关技术选择留待 Phase 7。

## 2. 功能详细设计统一模板

Task 6.2 和 Task 6.3 的每个功能必须依次包含以下 16 个章节，不得删除；不适用时必须写明“不适用”及原因。

1. 功能目的；
2. 使用角色；
3. 前置条件；
4. 页面入口；
5. 初始化与数据加载；
6. 字段与显示规则；
7. 按钮与操作；
8. 表单校验；
9. API 调用；
10. 状态变化；
11. 权限控制；
12. 成功反馈；
13. 异常处理；
14. 边界情况；
15. 日志与审计；
16. 验收标准。

每个章节必须引用对应 Approved/Frozen 依据，区分页面显示、客户端输入、服务端派生和数据库正式事实。

## 3. 页面初始化与生命周期

统一初始化顺序：

```text
身份校验
→ 页面访问权限校验
→ 加载字典和基础选项
→ 加载页面主数据
→ 加载辅助数据
→ 显示可操作页面
```

- 身份或页面权限失败时不得继续加载受保护数据；
- 主数据失败时显示页面级错误，不展示可误解为真实数据的空页面；
- 辅助数据失败不得伪装为正常，允许局部失败时必须标明失败区域并提供重试；
- 页面卸载、路由切换或请求被新请求替代时取消或忽略无效请求；
- 初始化只加载当前页面必需数据，不得一次加载全部业务数据；
- 重要查询条件应可从 URL 恢复，刷新后重新按正式权限加载；
- 页面恢复时不得依赖过期的业务状态或库存缓存。

## 4. 列表页统一规范

列表页统一支持查询、筛选、分页、排序、刷新、适用的导出与批量操作，并覆盖空状态、加载失败和权限范围。

| 参数 | 统一规则 |
| --- | --- |
| `page` | 从 1 开始，默认 1 |
| `pageSize` | 默认 20，最大 100 |
| `keyword` | 统一关键词参数 |
| `sortBy` | 使用 API 白名单字段 |
| `sortOrder` | 仅 `asc` 或 `desc` |

- 筛选条件变化后返回第一页；
- 查询、排序、刷新和导出均复用当前权限与数据范围；
- 必须区分系统无数据、当前筛选无结果、无权限查看和数据加载失败；
- 刷新保留有效筛选条件并获取服务端最新数据；
- 导出复用当前筛选、排序、字段权限及脱敏规则；
- 批量操作显示选中数量，并在执行前对每条记录重新校验权限、状态和数据范围；
- 部分记录不可操作时不得静默跳过，必须说明处理结果。

## 5. 详情页统一规范

详情页标准区域为：基本信息、业务明细、状态与进度、关联单据、附件、操作记录。

必须显示业务编号、当前状态、创建人和创建时间、更新人和更新时间、当前可执行动作、关键来源和关联单据。敏感字段、金额、成本和个人信息按字段权限展示或脱敏。

遵循 Phase 4 已批准布局：右侧信息区默认折叠。关联区域可局部加载，但失败时必须明确标识并允许重试，不得影响已成功加载的主数据。

## 6. 新增、编辑与只读模式

### 6.1 新增模式

- 加载 Approved/Frozen 允许的默认值和必要选项；
- 执行前端基础校验，提交后由服务端再次完整校验；
- 提交期间锁定按钮，防止连续点击和重复提交；
- 成功后读取服务端正式结果，不根据客户端草稿推测编号、状态或汇总。

### 6.2 编辑模式

- 进入时获取最新服务端数据；
- 校验当前状态是否允许编辑以及用户是否仍有权限；
- 使用 API 规定的版本、时间戳、幂等和并发规则；
- 不得静默覆盖其他用户的新修改；冲突时提示并重新加载最新数据。

### 6.3 只读模式

- 不显示编辑入口；
- 敏感字段按权限隐藏或脱敏；
- 状态动作与普通编辑分离，并在每次调用时独立校验权限、状态和版本。

## 7. 表单与字段校验

统一执行四层校验：格式校验、业务校验、关联对象校验、权限及数据范围校验。

格式校验覆盖必填、长度、类型、日期、数量、金额和小数精度。业务校验覆盖日期先后、数量和金额合法性、当前状态、来源业务前置条件及 Frozen 规则。

关联校验至少覆盖 SKU、仓库、店铺、厂家、来源单据和关联明细的存在性、有效性、归属、状态与数据范围。

前端校验只用于及时反馈，不能替代服务端校验。错误信息必须定位具体字段、明细行或业务问题，并提供可执行提示，禁止只显示“操作失败”。

## 8. 未保存内容保护

表单存在修改且尚未成功保存时，返回、关闭弹窗、切换页面、刷新或关闭浏览器标签页必须提示：

> 当前内容尚未保存，离开后修改将丢失，是否继续？

用户取消离开时保留输入；确认离开时清理草稿页面状态。只有服务端确认保存成功后才能解除拦截，保存失败或结果不确定时继续保护未保存内容。

## 9. 按钮与操作状态

每个按钮必须定义显示条件、启用条件、禁用原因、所需权限、当前业务状态、调用 API、成功结果和失败处理。

按钮统一状态为：隐藏、显示但禁用、可操作、执行中、执行成功、执行失败。

- 无功能权限时隐藏；有查看权限但业务条件不满足时可显示但禁用；
- 禁用原因必须以文字、提示或上下文说明，不能让用户猜测；
- 执行中锁定同一动作并防止重复操作；
- 执行成功后以服务端最新记录刷新按钮；
- 执行失败后结束 Loading、恢复合法操作状态并显示 Request ID。

## 10. 高风险操作确认

删除、作废、取消、撤销、反审核、库存调整执行、导入执行、导入重试、批量操作、附件删除、权限修改和敏感数据导出必须二次确认。

确认框必须包含操作对象、操作后果、是否可恢复，并在正式规则要求时填写原因。确认前和提交时均须复核权限、状态、版本及下游事实；确认框不得暗示客户端可以保证最终结果。

## 11. API 调用规范

所有功能必须使用状态为 Completed / Approved / Frozen 的 API Master Specification v1.1。

- 请求携带 `X-Request-ID`；适用写操作携带 `Idempotency-Key`；
- 前端不得自行修改、推测或缓存为正式状态；
- 前端不得直接连接或修改数据库；
- 写操作成功后获取服务端最新正式数据；
- 不得绕过正式 API，也不得通过页面逻辑创造新的数据库事实；
- URL、HTTP Method、请求、响应和错误码必须直接引用 Frozen API；
- 如功能无法由现有 API 完成，停止该部分并发起正式 Change Request，不得临时增加接口。

## 12. HTTP 异常处理

| HTTP 状态 | 页面处理 |
| --- | --- |
| 400 | 提示请求格式错误并保留 Request ID |
| 401 | 清理无效认证状态并进入登录页 |
| 403 | 显示无权限页面或操作提示 |
| 404 | 显示记录不存在、已删除或不可访问 |
| 409 | 显示数据、状态、幂等或并发冲突并重新加载 |
| 422 | 定位字段、明细或业务校验错误 |
| 429 | 提示操作频繁，结束 Loading，禁止立即重复提交 |
| 500 | 显示系统异常和 Request ID |
| 503 | 显示服务暂不可用并提供安全重试 |

不得向用户暴露 SQL、数据库结构、服务器路径、堆栈、Token、密钥或内部服务地址。未知错误不得展示原始服务端异常。

## 13. 状态操作规范

提交、审核、确认、撤销、作废等状态动作统一执行：

```text
锁定操作按钮
→ 调用正式状态动作 API
→ 等待服务端结果
→ 获取最新记录
→ 刷新状态和按钮
→ 刷新操作记录
```

禁止前端直接改变状态字段、仅凭成功 Toast 推测状态、让列表与详情显示不同状态，或在请求失败后保留虚假成功状态。服务端返回冲突时必须刷新最新状态和合法动作。

## 14. 权限控制规范

权限组合至少包括：身份、页面权限、功能权限、操作权限、记录范围、仓库范围、店铺范围、厂家范围、字段权限和敏感信息权限。

前端隐藏菜单和按钮仅用于交互体验，后端必须对每次请求重新验证权限。权限发生变化后，旧页面不得继续使用缓存权限完成操作；下一次请求被拒绝时必须清理失效能力并刷新页面权限状态。

列表、详情、汇总、导出、附件和关联跳转均必须先应用数据范围；无权数据不得通过数量、错误信息、搜索或排序被推断。

## 15. 成功提示、错误提示与反馈

- Toast：保存、提交、审核成功，导出任务创建及普通操作完成；
- Dialog：高风险或不可逆操作、离开未保存页面、冲突处理及需要用户选择的异常；
- 页面级状态：页面无法加载、无权限、记录不存在及服务不可用。

重要错误不得仅使用短暂 Toast。成功提示必须与服务端正式结果一致；失败提示包含安全原因、可执行下一步和 Request ID。

## 16. Loading 规范

区分页面级、区域级、按钮级 Loading 和异步任务状态。同一操作不得出现重复 Loading，Loading 期间不得重复提交，请求失败或取消后必须结束 Loading。

长时间任务不得无限显示无说明 Loading；导入、导出等长任务显示任务状态、进度或刷新入口，不阻塞整个页面。

## 17. 空状态与错误状态

每个列表和数据区域必须覆盖初始无数据、筛选无结果、无权限、加载失败、数据已失效和服务不可用。

空状态可提供“新建采购订单”“清除筛选条件”“重新加载”等合法下一步，但按钮仍须遵循功能、操作和数据范围权限。无权限不得伪装为无数据，失败不得伪装为空状态。

## 18. 并发、重复提交与幂等

必须处理连续点击、浏览器重复请求、网络重试、多用户同时修改、来源单据状态变化、库存余额变化和导入任务重复执行。

- 写操作按 API 规范使用幂等键；
- 冲突不得静默覆盖；
- 重复请求不得产生重复业务结果；
- 结果不确定时先查询正式结果，不能直接再次提交；
- 冲突后重新加载最新数据，并让用户基于新版本重新决定；
- 幂等键生成、保存和重试的具体技术实现留待 Phase 7。

## 19. 库存相关功能特别规则

所有库存变化必须由正式业务动作 API 触发。禁止页面直接编辑库存余额、普通 PATCH 修改库存数量、客户端计算并确认正式库存，或忽略库存并发冲突。

库存操作失败必须保持正式余额不变、流水不增加、来源单据累计不变化，页面显示服务端真实结果。页面不得通过乐观更新展示未经服务端确认的库存成功状态。

## 20. 导入与导出功能规范

### 20.1 导入

- 校验文件类型、大小、模板版本、表头和内容；
- 校验与执行分离；
- 展示成功、失败、部分成功和失败明细；
- 使用幂等与任务状态防止重复执行；
- 海外仓库存只允许通过 Approved/Frozen Excel 导入流程更新；
- 结果以服务端正式任务、流水及余额为准。

### 20.2 导出

- 使用用户当前权限、查询条件和数据范围；
- 敏感字段执行字段权限和脱敏；
- 大数据导出采用异步任务；
- 导出动作记录审计日志；
- 下载时重新校验权限。

## 21. 附件功能规范

附件功能覆盖上传、列表、预览或下载、关联、解除关联、删除及生命周期。

- 校验文件类型、大小、MIME 和内容特征；
- 不暴露真实存储路径或凭证；
- 每次预览或下载重新验证对象和字段权限；
- 已作为正式业务证据的附件不得随意解除或删除；
- 敏感附件访问、解除和删除必须记录日志；
- 只使用 Frozen Attachment API，不建立模块专用附件事实。

## 22. 日志与审计规范

登录、退出、新增、修改、删除、提交、审核、确认、撤销、反审核、作废、库存变化、导入、导出、附件下载和删除、用户/角色/权限调整及敏感数据访问必须按 API 规范记录。

日志至少关联操作人、操作时间、动作、对象类型、对象 ID 或编号、执行结果、Request ID 和安全变更摘要。页面不得记录密码、Token、完整敏感值、SQL、路径或堆栈；日志展示和导出继续执行权限及脱敏。

## 23. 性能与数据加载规范

- 所有大列表分页；
- 大型下拉选项使用远程搜索；
- 搜索输入使用合理防抖；
- 页面只请求当前需要的数据，避免重复请求；
- 字典等低风险数据可短期缓存；
- 库存和业务单据状态不得依赖过期缓存；
- 大文件导入、导出采用异步任务。

具体缓存、请求协调、异步任务和防抖技术留待 Phase 7，不在 Phase 6 固定框架或第三方库。

## 24. 可访问性与易用性

- 表单控件具有明确标签；
- 错误信息可定位到字段、明细或业务区域；
- 键盘可完成基础操作；
- 状态不得仅依赖颜色表达；
- 禁用状态具有文字原因；
- 危险操作具有明确警示；
- 表格重要信息在常见桌面宽度下可阅读；
- 焦点、Loading、Dialog 和错误反馈顺序清晰。

## 25. 功能验收标准

每个后续功能至少验证：

1. 正常流程可完整完成；
2. 状态变化与 Frozen 状态机一致；
3. 权限不足不能操作；
4. 数据范围不能越权；
5. 重复提交不产生重复结果；
6. 错误提示明确；
7. 列表与详情数据一致；
8. 页面刷新后正式结果不丢失；
9. 日志记录完整；
10. 关联数据和来源追溯正确；
11. 并发冲突不会静默覆盖；
12. 不违反 Frozen 业务、数据库、页面和 API 规范。

Task 6.2 与 Task 6.3 必须在各功能的第 16 章将上述标准细化为可验证条件，不得以“符合规范”代替具体结果。

## 26. Phase 6 边界与变更控制

Phase 6 采用三个 Task 加独立 Final Consistency Review：Task 6.1 定义统一规则，Task 6.2 定义核心业务功能，Task 6.3 定义公共能力；Final Consistency Review 是 Phase Exit Gate，不作为普通 Task。

本规范不得修改 Frozen Database Logical Design v1.1 或 API Master Specification v1.1，不得新增业务模块、状态、字段、表、关系或接口。后续发现缺口时必须登记 DCR 或 Change Request；技术框架、代码结构和具体实现工具统一留待 Phase 7。
