# Task 6.2：核心业务功能详细设计（Core Business Functional Design）

## 1. 文档目的与边界

本文集中定义基础资料、采购管理、生产与质量验收、库存管理、出入库与调拨、跨境业务六个核心业务模块如何运行。

所有功能统一引用 [Task 6.1 功能详细设计统一规范](TASK_6_1_FUNCTIONAL_DESIGN_STANDARD.md) 的 16 章模板。页面初始化、分页、Loading、Toast、HTTP 异常、通用权限、未保存保护、幂等和可访问性均直接适用 Task 6.1，本文只描述模块差异。

本文不重新定义业务模块、Approved 页面、Frozen 数据库或 API。Frozen 业务规则、Database Logical Design v1.1、API Master Specification v1.0 与 Approved Phase 4 页面优先。若功能无法映射现有 API，必须停止冲突部分并登记 Change Request，不得临时增加字段、状态、接口或页面事实。

本文不指定前端框架、后端框架、ORM、组件库或第三方技术；技术选型留待 Phase 7。不创建真实页面、API Route 或业务代码。

## 2. 采购与生产平行原则

采购管理与生产管理是平行业务模块，直接采购与委外生产分别独立运行。采购单不生成生产单，生产单不要求采购单作为前置条件，也不建立页面、数据库或 API 的直接关系。

生产订单只使用 `PRO-003` 已冻结字段独立创建，不读取采购订单标识，不校验采购订单状态，不通过采购业务限制生产计划数量，也不保存目标仓库。完工后目标仓库只由分批完工记录确定。

两条业务流程在质量验收节点汇合：一张验收单只能选择采购来源或生产来源，二者互斥。验收确认只形成后续入库资格，不直接修改库存。

```text
直接采购流程 ─┐
              ├→ 质量验收 → 正式入库
委外生产流程 ─┘
```

禁止通过临时字段、隐藏字段、JSON 扩展、备注、客户端缓存、编号文本、非正式关联表或新查询/响应字段建立隐性采购—生产关系。

## 3. 模块一：基础资料

### 3.1 功能目的与范围

基础资料为采购、生产、库存、出入库和跨境业务提供唯一引用来源。覆盖产品、SKU、配件、分类、品牌、供应商、生产厂家、仓库、销售平台、店铺、客户快照及 Approved 的其他资料。

配件按产品与 SKU 正式模型管理，不建立平行配件对象。客户资料只查询销售出库客户快照，不建立独立客户主数据。

### 3.2 页面与运行规则

- 列表支持查询、筛选、排序、分页、查看及适用导出；
- 有权限的正式资料支持新增、编辑、启用和停用；
- 新增、编辑前加载最新数据与有效选项；
- 编码、名称和唯一字段由服务端最终校验；
- 已停用资料不得进入新业务选项，历史单据继续显示其快照；
- 已被正式业务引用的资料不得物理删除；
- 并发冲突不得静默覆盖，重新加载后由用户决定；
- 产品是业务分类与公共信息主体，SKU 是单据和库存使用的正式数量粒度；
- 产品停用后其资料不得用于新业务，SKU 是否可选仍按自身状态及正式规则校验；
- 页面不得创造 Frozen 数据库中不存在的字段。

### 3.3 客户及敏感信息

客户姓名、地址、联系方式、供应商/厂家联系及银行税务字段按字段权限返回或脱敏。列表、详情、关键词搜索和导出均在服务端执行字段及数据范围权限，无权限用户不得通过数量、搜索命中、排序或错误信息推断原值。

### 3.4 权限与范围

仓库资料叠加仓库范围，店铺和客户快照叠加店铺范围，厂家资料及厂家仓叠加厂家与仓库范围；金额、价格和敏感字段另行校验。前端入口不替代服务端授权。

| 页面/功能 | 用户动作 | 前置状态 | API 编号 | 成功结果 | 失败处理 |
| --- | --- | --- | --- | --- | --- |
| 正式资料列表 | 查询、筛选、分页、排序 | 有查看权限 | 各 `MD-*-01` | 返回授权列表 | 区分无数据、无权限与加载失败 |
| 正式资料详情 | 查看 | 记录可见 | 各 `MD-*-02` | 返回允许字段 | 404/403 按 Task 6.1 |
| 正式资料新增 | 保存 | 有新增权限、引用有效 | 各 `MD-*-03` | 创建启用资料 | 唯一/校验错误定位字段 |

## 4. 模块二：采购管理

### 4.1 功能范围与独立性

采购模块覆盖采购订单、明细、付款、退货、采购进度、采购来源质量验收、采购入库及关联记录。采购订单生命周期不包含生成、选择或关联生产单，也不通过生产业务计算采购状态。

采购进度只由采购订单自身状态、采购付款、采购来源验收、采购入库及采购退货等 Frozen 事实汇总。

### 4.2 采购订单新增与编辑

新增时选择有效供应商、采购类型、SKU、数量、单价、交期、币种及 Approved 业务归属；适用附件使用通用附件功能。数量大于零、金额和精度合法、交期关系合法，供应商与 SKU 启用且用户有记录及金额权限。

草稿或已驳回且正式规则允许时编辑。提交后普通字段不得随意修改；每次编辑先读取最新数据并携带版本。成功后重新读取服务端正式金额、状态和快照。

### 4.3 状态动作

提交、撤回、审核、驳回、反审核、取消和作废全部调用 `PUR-*` 正式动作 API。已有付款、验收、入库、退货或其他下游事实时，反审核、作废及回退按 Frozen 条件受限。

### 4.4 采购付款

付款必须关联合法采购订单，校验币种、金额、累计金额、付款权限及敏感字段权限。付款进度由正式付款记录汇总，不改变采购完成状态；已创建付款事实不得直接删除。

### 4.5 采购退货

退货必须关联原采购、原入库及合法明细，数量不得超过可退数量。退货单创建、审核本身不减少库存；实际库存变化仅由 `OUT-017` 确认采购退货出库产生。

### 4.6 采购来源质量验收

创建验收单时使用 `sourceType=purchase`，只选择合法采购来源。采购来源和生产来源互斥；校验采购状态、可验收量、验收仓和权限。确认验收形成采购入库资格，不直接增加库存。

### 4.7 页面动作与 API 映射

| 页面/功能 | 用户动作 | 前置状态 | API 编号 | 成功结果 | 失败处理 |
| --- | --- | --- | --- | --- | --- |
| 采购列表/详情 | 查询、查看 | 记录及金额权限 | `PUR-001`、`PUR-002` | 返回授权采购数据 | 403/404/加载失败分开处理 |

| 采购审核 | 通过/驳回 | 待审核、职责分离 | `PUR-007`、`PUR-008` | 写审批与状态历史 | 权限/版本失败不展示成功 |

## 5. 模块三：生产与质量验收

### 5.1 生产订单独立创建

生产订单独立创建，只选择生产厂家及 `PRO-003` 已定义的日期、币种、备注和 SKU 计划明细。不要求采购单作为前置，不读取或展示采购订单标识，不校验采购状态，不通过采购业务限制生产数量，也不建立直接页面跳转。

生产订单不保存目标仓库。页面不得通过隐藏字段、备注或缓存保存目标仓；目标仓仅在分批完工记录中选择。

### 5.2 生产执行流程

```text
独立创建生产订单
→ 提交
→ 审核
→ 开始生产
→ 登记进度
→ 分批完工
→ 生产来源质量验收
→ 生产入库或按 Approved 厂家仓流程处理
```

生产订单的状态、审核和执行进度相互按 Frozen 规则展示。进度记录是过程事实，不替代分批完工、验收、入库或库存流水。

### 5.3 分批完工与目标仓库

分批完工严格使用 DCR-001：Draft、Confirmed、Revoked、Voided。

- Draft 可创建并按正式可编辑范围处理；
- Confirmed 不得普通编辑；
- Confirmed 仅在无验收、入库、发货、库存流水等下游时撤销；
- Draft 可在正式条件下作废，Voided 不恢复；
- 确认、撤销和作废后重新读取服务端正式结果；
- 目标仓库只存在于分批完工记录，必须是授权且适用的公司仓、对应厂家仓或 Approved 合法仓库语义；
- 不得回写生产订单。

### 5.4 生产来源质量验收

创建验收时使用 `sourceType=production`，只选择合法生产来源；不得同时提供采购来源。校验生产来源状态、合法完工累计、可验收数量、验收仓、验收人和厂家/仓库范围。

验收数量满足验收数量等于合格数量加不合格数量。验收人员可以不是仓库人员，但必须具有验收及对应记录/仓库权限。确认只形成后续入库资格；不合格部分不得进入正式入库。

撤销确认受入库等下游事实限制。不建立独立生产异常对象，异常仅使用现有进度、验收结果、缺陷、说明和跟踪信息。

### 5.5 委外加工付款

生产付款关联合法生产订单与厂家，执行金额、币种、累计、状态、权限及敏感字段校验。付款记录不改变生产进度、完工、验收或库存事实，已形成付款事实不直接删除。

## 6. 模块四：库存管理

### 6.1 库存查询与来源

当前库存按仓库、SKU、产品、分类及 Approved 状态查询；余额与流水分开展示，流水可跳转正式来源单据。仓库权限在筛选和聚合前执行，页面只分页加载当前所需数据。

正式库存变化仅可来源于采购入库、生产入库、销售出库、采购退货出库、调拨、跨境发运、库存调整及海外仓库存 Excel 导入。

禁止直接编辑余额、普通 PATCH 修改数量、客户端计算并确认正式库存，以及删除或覆盖库存流水。

### 6.2 库存调整

```text
创建调整单
→ 填写原因和调整明细
→ 提交
→ 审核
→ 执行
→ 原子更新余额、流水、状态和审计
```

调整原因必填，方向、数量、成本和结果合法。执行前重新校验余额、版本、仓库权限、来源和防负库存条件；失败时余额、流水、累计和状态整体不变；重复执行不得产生重复流水。

### 6.3 库存预警

库存预警仅用于提醒，可跳转合法采购、生产或库存业务入口，但不自动创建单据、不改变库存事实。查看、处理和关闭受仓库及记录范围控制。

### 6.4 并发与边界

多用户操作、来源状态变化、库存余额变化、重复提交和网络重试均由幂等键、版本、余额条件更新及事务处理。冲突后重新加载余额和来源；禁止静默覆盖或负库存。

## 7. 模块五：出入库与调拨

### 7.1 入库

入库必须选择 Approved 合法来源。采购入库与生产入库分别使用对应验收资格；生产来源不合格数量不得入库。同一来源累计不得超过合法可入库量。

确认入库在单一事务中增加目标仓余额、追加流水、更新来源累计、单据状态和审计；任何一步失败均整体回滚。冲销使用反向流水，不删除历史。

### 7.2 国内销售及其他出库

本系统不管理完整销售订单。国内电商及线下销售逐单登记销售出库，选择合法店铺、仓库、SKU、批次、数量及 Approved 字段。确认前重新校验库存和权限，成功时原子扣减库存并追加流水。

其他出库只用于样品、领用及其他 Approved 类型，不替代调拨、采购退货、跨境或报损流程。

### 7.3 采购退货出库

采购退货出库必须关联已审核采购退货及原入库事实，数量不超过合法可退量。`OUT-017` 确认后原子扣减库存并更新采购退货完成事实；采购退货页面本身不得直接减少库存。

### 7.4 调拨

```text
创建调拨单
→ 提交与审核
→ 调出仓确认
→ 进入在途仓
→ 调入仓确认
→ 增加目的仓库存
```

来源、在途、目的仓必须符合 Frozen 类型并两两不同。调出与调入分阶段，调入累计不得超过合法在途量；取消或冲销不得删除历史库存事实。每阶段使用双仓原子事务、幂等和防负库存校验。

## 8. 模块六：跨境业务

### 8.1 跨境发货流程

```text
选择合法来源库存
→ 创建跨境发货
→ 提交与审核
→ 确认发运
→ 减少来源仓库存
→ 增加在途仓库存
→ 等待海外仓库存导入
```

来源可以是合法厂家仓、公司仓或 Approved 产业园成品仓语义；在途仓与目的海外仓按 Frozen 仓库类型校验。关联生产时只使用 Frozen 生产单/明细关系，不关联采购单或完工记录 ID。

### 8.2 正式状态与发运确认

跨境发货只使用 `status`、`approvalStatus`、`shipmentStatus`。不得重新引入海外收货、独立物流扩展、差异处理状态或手工海外收货。

发运确认校验来源仓、在途仓、目的仓、明细、批次、库存、状态和权限；在一个事务中减少来源库存、增加在途库存、追加双向流水、更新发运累计与审计。重复确认不得重复扣减。

### 8.3 海外仓库存导入

```text
获取正式模板
→ 上传 Excel
→ 校验模板与数据
→ 查看错误和警告
→ 执行导入
→ 匹配跨境发货及在途来源
→ 减少在途库存
→ 增加海外仓库存
→ 生成流水与来源追溯
```

海外仓库存唯一正式来源为 Excel 导入，不允许手工增加或手工海外收货。重复文件、重复任务或成功行重试不得重复增加库存。部分成功明确成功行、失败行和原因，正式结果可追溯至发货、导入任务、匹配和库存流水。

Task 5.4 的 `CBR-018`—`CBR-020` 是跨境业务专用只读投影；Task 5.5 的 `IMP-*` 是通用导入创建、模板、校验、执行、重试、结果和历史接口。只读投影不得承担写入，通用导入不得绕过跨境匹配规则。

## 9. 六模块统一 16 章模板

本节按 Task 6.1 强制模板逐章覆盖六个模块；各章的模块细节、动作和 API 以第 3 至 8 节为准。

### 9.1 功能目的

基础资料提供唯一引用来源；采购管理独立处理直接采购；生产与质量验收处理委外生产及两类互斥来源验收；库存管理维护余额、流水、预警和调整；出入库与调拨承接正式库存事务；跨境业务处理发运、在途和海外仓 Excel 导入。

### 9.2 使用角色

基础资料管理员使用基础资料功能；采购、财务及采购审核角色使用采购功能；生产、验收及生产审核角色使用生产与验收功能；仓库、库存审核及库存管理角色使用库存、出入库和调拨功能；跨境、导入及跨境审核角色使用跨境功能。所有角色同时受记录、厂家、仓库、店铺、金额、成本和敏感字段范围约束。

### 9.3 前置条件

六个模块均要求用户已认证且具有功能及数据范围权限，相关主数据有效，来源对象存在且状态合法；采购与生产分别独立创建，验收来源互斥；库存写入另要求来源资格、余额、版本、仓库及幂等条件全部通过。

### 9.4 页面入口

页面入口沿用 Approved Phase 4 导航：基础资料、采购管理、生产管理、质量验收、库存管理、出入库、调拨及跨境业务。禁止增加采购与生产之间的直接创建入口、关联区域或隐性跳转。

### 9.5 初始化与数据加载

各模块按 Task 6.1 顺序加载身份、页面权限、字典/选项、主数据和辅助数据；列表使用服务端分页，详情只加载当前记录的正式关联。生产页不加载采购标识或采购状态，采购页不加载生产关系。

### 9.6 字段与显示规则

基础资料显示授权字段和历史快照；采购显示订单、付款、退货、采购验收及入库事实；生产显示厂家、计划、进度、完工、生产验收及入库事实；库存显示余额、流水、预警和调整；出入库/调拨显示来源与库存事务；跨境显示三正式状态、物流信息、在途、海外库存及导入追溯。所有字段均来自 Approved/Frozen 定义。

### 9.7 按钮与操作

基础资料提供查询、维护和启停；采购与生产分别提供各自创建、编辑、状态及业务动作；验收提供创建、提交、确认、撤销和作废；库存调整、入库、出库、调拨和跨境按正式状态显示审核及执行按钮。按钮矩阵见第 11 节。

### 9.8 表单校验

六个模块统一校验必填、格式、精度、唯一性、引用有效性、状态、数量、金额、版本和权限；采购/生产验收额外校验来源互斥与可验收量；库存相关表单额外校验仓库类型、来源资格、可入/出/调/发数量及防负库存。

### 9.9 API 调用

基础资料仅调用 `MD-*`；采购仅调用 `PUR-*` 及采购来源 `INS-*`；生产仅调用 `PRO-*` 及生产来源 `INS-*`；库存调用 `INV-*`；出入库和调拨调用 `INB-*`、`OUT-*`、`TRF-*`；跨境调用 `CBR-*` 与 `IMP-*`。详细映射见各模块表，不新增接口或参数。

### 9.10 状态变化

资料启停、采购、生产、DCR-001 完工、验收、库存调整、入库、出库、调拨和跨境状态均由对应动作 API 按 Frozen 状态机改变；客户端不提交目标状态。只读查询、进度、付款、附件、导出和日志不得伪造业务状态变化。

### 9.11 权限控制

服务端逐请求执行功能、操作、记录及字段权限，并按厂家、仓库、店铺等范围过滤；金额、成本、客户个人信息及联系/银行税务字段单独授权或脱敏。前端隐藏、禁用或路由控制不替代服务端校验。

### 9.12 成功反馈

查询成功显示正式结果；保存成功刷新服务端版本；状态及库存动作成功刷新状态、累计、余额和流水；导入成功显示任务及行级结果。Toast 文案区分“已保存”“已提交”“已确认/执行”，不得在结果未知时宣告成功。

### 9.13 异常处理

六个模块统一处理 400、401、403、404、409、422、429 和 5xx；字段错误定位到表单或明细行，权限/不存在/冲突分开呈现，库存或导入失败不得保留部分业务事实。具体规则见第 12 节及 Task 6.1。

### 9.14 边界情况

覆盖资料停用、来源失效、重复提交、并发修改、状态变化、下游阻塞、数量/金额超限、库存不足、网络中断、部分导入成功和页面刷新失败。采购与生产任何隐性直连均不作为容错方案。

### 9.15 日志与审计

资料维护、敏感查看、审批、状态动作、付款、验收、完工、库存执行、入出库、调拨、发运、导入、导出和权限拒绝按 Task 6.1 与 API Master Specification 记录 Request ID、操作人、对象、结果及必要前后值；日志不得保存未脱敏秘密或文件内容。

### 9.16 验收标准

六个模块必须通过第 13 节验收：页面动作均映射 Frozen API，状态、权限、来源、库存和日志规则一致；采购与生产保持平行且验收来源互斥；无新增模块、对象、字段、关系、状态或接口，无代码和依赖变化。

## 12. 异常与边界情况

六个模块均按 Task 6.1 处理以下情形：

- 重复提交：复用幂等键查询首次结果，不产生重复事实；
- 并发修改或状态已变化：返回 409，重新加载最新记录和按钮；
- 权限已变化：停止操作、清理旧能力并重新授权页面；
- 关联资料已停用：新业务拒绝，历史快照继续显示；
- 来源不存在或状态不允许：停止写入并定位来源问题；
- 下游已产生：禁止反审核、撤销、作废或普通修改；
- 数量或金额超限：422 定位字段或明细行；
- 库存不足：事务不写余额、流水、累计或状态；
- 网络中断或结果不确定：先查询正式结果，不直接重试写入；
- 写成功但页面刷新失败：提示写入可能已完成，保留 Request ID 并提供重新加载，不重复提交。
