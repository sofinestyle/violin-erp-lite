# Task 6.3：公共能力功能详细设计（Common Capability Functional Design）

## 1. 文档目的与边界

本文集中定义 Violin ERP Lite 的公共能力如何运行，覆盖登录与会话、用户与权限、Excel 导入、附件、导出、日志与审计、系统设置、消息提醒、编号、数据字典、打印、国际化、系统时间以及统一查询、状态动作、异常和删除原则。

本文直接引用 [Task 6.1 功能详细设计统一规范](TASK_6_1_FUNCTIONAL_DESIGN_STANDARD.md)，并与 [Task 6.2 核心业务功能详细设计](TASK_6_2_CORE_BUSINESS_FUNCTIONAL_DESIGN.md) 共同构成 Phase 6 功能详细设计。本文只定义已批准能力如何运行，不新增独立业务模块、数据库对象、状态、页面、API、请求字段或响应字段。

Frozen BUSINESS_RULES、Database Logical Design v1.1、API Master Specification v1.1 和 Approved Phase 4 页面设计优先。技术框架、认证协议、Token/Session 存储、文件存储、异步任务实现、缓存和中间件留待后续技术阶段。

## 2. 公共能力统一原则

凡属于公共能力的功能，包括登录、权限、编号、附件、导入、导出、日志、审计、查询、状态动作、异常提示、数据字典、打印、国际化和系统时间，必须统一设计、统一实现、统一维护。业务模块不得重复实现或派生独立版本；确需扩展时，只能在现有公共能力基础上扩展，不得破坏统一行为。

公共能力必须遵守以下边界：

- 统一身份、权限、数据范围和字段权限；
- 统一 Header、请求响应、分页、排序、筛选、错误码、幂等和审计；
- 统一文件、任务、时间、编号和展示语义；
- 页面显示和客户端输入不得成为数据库正式事实；
- 无独立 Frozen API 的能力只定义产品规则或复用现有业务 API，不虚构接口；
- Phase 7 实现若无法映射现有 Frozen 内容，必须停止冲突部分并走正式变更流程。

## 3. 登录与会话

### 3.1 统一身份

系统只有一套用户身份和权限体系。PC 管理端与微信小程序最终都映射到 Frozen `users`、`user_roles`、`roles`、`role_permissions`、`permissions`、`role_warehouses` 和 `role_stores`，使用相同角色、权限、数据范围及字段权限。

不得建立独立微信用户体系。微信昵称和头像仅可作为客户端展示信息，不能替代系统用户身份、用户名、角色或审计操作人。

### 3.2 PC 管理端登录

PC 管理端使用用户名和密码登录，调用 `SEC-001`。页面不得保存、回显、记录或通过 URL 传递明文密码；登录成功后通过 `SEC-004` 获取当前会话摘要，通过 `SEC-005` 获取当前界面能力摘要。

连续失败按 Frozen `failed_login_count`、`locked_until` 语义和统一安全策略限制。提示不得暴露用户名是否存在、密码哈希、锁定实现或内部风控细节。

### 3.3 微信小程序登录与绑定

微信小程序支持微信授权登录。首次使用微信授权时，必须绑定一个已存在且启用的系统账号；绑定过程必须验证该系统账号的合法凭据和当前状态。绑定成功后，后续可通过微信授权自动登录，并始终映射回同一个系统用户身份。

Phase 6 不定义微信身份映射的持久化介质，不新增或指定 `openid`、`unionid`、Token、Refresh Token、Session 或客户端来源数据库字段，不指定 OAuth、JWT、Redis 或认证中间件。后续技术方案必须在不改变 Frozen 数据库和 Frozen API 的前提下实现；若无法实现，必须停止并走正式变更流程。

微信首次绑定和后续授权均归入 `SEC-001` 的统一登录产品流程，不在本文新增请求参数、路径或接口编号。

### 3.4 会话生命周期

- 刷新认证上下文使用 `SEC-002`，必须防重放；
- 主动退出使用 `SEC-003`，成功后当前认证上下文失效；
- 当前会话使用 `SEC-004` 查询，不返回密码、Token 原值或内部存储信息；
- 当前权限摘要使用 `SEC-005`，仅供界面展示，不能替代业务 API 实时校验；
- 收到 401 时清理失效登录态并进入登录页，不自动重试写操作；
- 用户被停用后，不得通过 PC、微信小程序或旧会话继续登录或操作；
- 角色、权限或数据范围变化后，下一次请求立即按最新授权判断，旧页面清理失效按钮、菜单及缓存数据；
- PC 和微信小程序可以同时存在合法会话，但任一客户端均不得获得额外权限。

### 3.5 登录安全与日志

登录成功、失败、刷新、登出、锁定、停用拒绝和异常风险事件按统一日志规则记录。日志区分 `pc`、`wechat-mini-program` 等客户端来源，但客户端来源只用于追溯，不影响身份和权限判断。

日志不得记录密码、Token、刷新凭证、授权临时凭据、Cookie、密钥或完整敏感数据。未知用户失败事件如无法合法映射 Frozen 对象，不得伪造对象标识，其安全遥测实现留待技术阶段。

## 4. 用户、角色与权限

### 4.1 用户管理

Approved 用户管理入口覆盖用户列表、详情、新增、编辑、启用和停用：

- 列表按用户名、显示姓名、状态及现有安全字段进行授权查询、筛选、排序和分页；
- 详情显示基本信息、账号状态、有效角色、数据范围和安全摘要，密码哈希永不返回；
- 新增校验用户名唯一、显示姓名、联系方式格式、初始角色及操作权限；
- 编辑不得直接修改创建/审计字段，不得将密码写入普通日志；
- 启用前重新校验账号资料和角色有效性；
- 停用为高风险操作，必须说明影响、二次确认并审计，历史业务及日志继续保留；
- 用户不能编辑自身角色、权限、数据范围或账号状态来提升权限；
- 系统最后一个必要管理身份不得被无保护地停用，具体职责约束以正式权限规则为准。

用户维护映射 Frozen `users` 和既有关系。API Master Specification v1.1 未登记独立用户管理 API，因此本文只定义 Approved 页面产品行为，不虚构接口编号、URL、请求或响应；后续实现必须先确认正式 API 覆盖。

### 4.2 角色管理

角色列表和详情展示角色编码、名称、说明、系统角色标识、启停状态、用户数量及授权摘要。新增、编辑、启用、停用必须校验角色编码唯一、操作者权限、职责分离及正式引用。

系统角色不得由普通管理员破坏性修改。角色停用不删除历史关系，停用后不得授予新用户；已登录用户的有效权限在下一次请求按最新有效角色重新计算。

角色维护映射 Frozen `roles`，API Master Specification v1.1 未登记独立角色管理 API，本文不新增接口。

### 4.3 权限分配

权限分配统一以角色为主体：

```text
选择角色
→ 加载功能与操作权限
→ 加载仓库、店铺及适用业务范围
→ 比较变更前后差异
→ 高风险确认
→ 服务端重新校验操作者授权
→ 保存并审计
→ 受影响用户后续请求按新权限生效
```

- 用户与角色通过 `user_roles` 维护有效期，不建立用户直接权限表；
- 角色与权限通过 `role_permissions` 维护；
- 仓库范围通过 `role_warehouses` 及 `access_level` 维护；
- 店铺范围通过 `role_stores` 及 `access_level` 维护；
- 厂家范围按现有厂家仓、业务记录和正式来源关系派生，不新增角色厂家范围表；
- 金额、成本、个人信息和敏感字段通过现有权限编码及字段权限规则控制，不新增平行字段权限对象；
- 权限管理员不能授予自身不具备或无权管理的权限；
- 制单、审核、仓库执行和高风险调整继续遵守职责分离。

权限目录和分配映射 Frozen `permissions`、`role_permissions` 及数据范围关系。`SEC-005` 只查询当前用户能力摘要，不承担管理写入。API Master Specification v1.1 未登记用户/角色/权限管理写接口，本文不虚构接口。

### 4.4 权限生效与审计

每次 API 请求都重新验证身份、功能、操作、角色有效性、记录、仓库、店铺、厂家派生范围及字段权限。前端隐藏菜单或按钮不能替代服务端校验。

新增/停用用户、角色调整、权限授予/撤销、数据范围变化和系统角色操作必须记录操作者、时间、目标、变更前后摘要、结果和 Request ID。日志不得记录密码或完整敏感值。

## 5. Excel 导入

### 5.1 正式流程

```text
上传文件
→ 校验文件和模板版本
→ 字段及行级业务校验
→ 预览校验结果
→ 用户确认正式执行
→ 执行并查看结果
```

校验与正式执行必须分离。未完成校验或存在阻断错误时，不得执行。

### 5.2 页面与接口

| 功能 | Frozen API | 产品行为 |
| --- | --- | --- |
| 模板下载 | `IMP-006`、`IMP-007` | 获取正式模板及可用版本 |
| 上传文件 | `IMP-001` | 创建导入任务并上传 Excel |
| 模板兼容校验 | `IMP-008` | 校验列、版本和文件特征 |
| 行级校验 | `IMP-009` | 启动字段、数据、唯一、外键、状态和权限校验 |
| 校验预览 | `IMP-010` | 分页查看错误、警告和行号 |
| 正式执行 | `IMP-011` | 二次确认后幂等执行 |
| 失败行重试 | `IMP-012` | 仅重试合法失败行，不重复成功行 |
| 结果查看 | `IMP-004`、`IMP-013` | 查看任务状态、成功、失败和部分成功结果 |
| 历史与导出 | `IMP-014`、`IMP-015` | 按权限查询和导出历史/结果 |
| 取消 | `IMP-005` | 仅在 Frozen 允许状态取消 |

### 5.3 校验、执行与追溯

- 文件扩展名、MIME、内容特征、大小、工作表、表头和恶意内容必须校验；
- 模板版本、必填列、类型、长度、精度、日期、编码、文件内重复及正式唯一范围必须校验；
- SKU、仓库、店铺、来源单据、状态和权限执行行级校验；
- 校验结果显示总行数、通过、失败、警告、行号、字段及安全错误原因；
- 执行使用幂等键和最新任务版本，重复提交不得重复生成业务事实；
- 部分成功必须明确成功行、失败行及结果对象，不静默忽略；
- 合法失败行重试继续使用原任务和原行，成功行不得再次执行；
- 文件、任务、模板版本、操作人、时间、目标范围、校验和执行结果完整追溯；
- 原文件名不得作为真实存储路径，页面不得暴露内部文件引用。

海外仓库存导入严格遵守 Task 6.2：Excel 导入是海外仓库存唯一正式来源，执行必须匹配跨境发货和在途库存，原子减少在途仓、增加海外仓并追加流水；不得手工海外收货或手工增加海外库存。

## 6. 附件管理

### 6.1 统一生命周期

所有业务模块复用 `ATT-001` 至 `ATT-008` 和 Frozen `attachments`、`attachment_links`：

```text
选择文件
→ 校验类型、大小、MIME、内容和权限
→ 上传附件
→ 关联合法业务对象
→ 查询、预览或下载
→ 按正式证据规则解除关联、失效或删除
```

### 6.2 功能规则

- 上传使用 `ATT-001`；上传失败不得形成虚假附件记录或正式关联；
- 查询和详情使用 `ATT-002`、`ATT-003`；
- 预览仅对允许类型复用 `ATT-003` 的安全元数据和 `ATT-004` 的重新鉴权下载结果，不新增预览 API；
- 下载使用 `ATT-004`，每次重新校验对象、数据范围和敏感附件权限；
- 关联使用 `ATT-005`，校验对象存在、状态和编辑权限；
- 解除关联使用 `ATT-006`，不得破坏正式业务证据；
- 删除或失效使用 `ATT-007`，仅无正式历史引用且策略允许时执行；
- 生命周期使用 `ATT-008` 查询，由附件、关联和审计记录派生；
- 正式业务证据不得随意物理删除；
- 原文件名不得作为真实存储路径，响应不得暴露内部存储地址、凭证或密钥；
- 上传、敏感预览/下载、关联、解除和删除均按风险等级审计。

## 7. 导出能力

### 7.1 统一导出规则

导出统一复用当前筛选、排序、字段权限、记录范围及脱敏规则，覆盖：

- 当前筛选结果导出；
- 单据及业务明细导出；
- 导入结果导出 `IMP-015`；
- 日志导出 `LOG-004`；
- 各模块已冻结的 `PUR-*`、`PRO-*`、`INS-*`、`INV-*`、`INB-*`、`OUT-*`、`TRF-*`、`CBR-*` 导出接口。

不得自行增加 PDF 导出。仅当 Approved/Frozen 页面已有打印需求时按第 13 节处理，打印不等同新增 PDF 导出。

### 7.2 异步、下载与历史

- 大数据导出按 Frozen API 的响应契约使用异步结果，不阻塞页面；
- 页面展示接口正式返回的任务状态、完成/失败结果和下载入口，不自建导出状态；
- 下载前再次验证用户、功能、数据范围及敏感字段权限；
- 下载文件仅在正式返回的有效期内可用，过期后不得绕过权限恢复；
- 导出历史从统一 `audit_logs` 及现有导出结果查询，不建立独立导出对象或表；
- 失败时显示安全原因和 Request ID，结果不确定时先查询正式结果；
- 每次导出和敏感下载均记录查询条件摘要、范围、字段、结果和操作者。

API Master Specification v1.1 未定义独立通用导出任务管理 API。本文不新增导出任务接口、状态或页面；任务状态、下载和有效期只使用各 Frozen 导出接口的正式返回或属于统一产品规则。

## 8. 日志与审计

### 8.1 六类统一查询分类

登录日志、操作日志、审计日志、导入日志、导出日志和安全日志统一复用 Frozen `audit_logs` 及适用导入对象。六类日志只是统一日志来源的查询分类，不建立平行日志表、对象或状态。

| 分类 | 主要事件 |
| --- | --- |
| Login Log | 登录成功/失败、刷新、退出、停用拒绝 |
| Operation Log | 新增、编辑、状态动作和业务执行 |
| Audit Log | 高风险操作、审批、权限及敏感访问 |
| Import Log | 上传、校验、执行、重试和结果 |
| Export Log | 导出条件、范围、结果和下载 |
| Security Log | 权限拒绝、限流、重放和风险事件 |

### 8.2 查询、导出与保护

- 列表使用 `LOG-001`，详情使用 `LOG-002`，链路追踪使用 `LOG-003`，导出使用 `LOG-004`；
- 支持时间、用户、模块、动作、对象、结果、Request ID、Trace ID 及现有链路字段筛选；
- 日志只追加，不允许普通用户创建、修改或删除；
- 产品层永久保留正式审计历史，不提供删除入口；归档实现不得破坏 Frozen 历史保护；
- 查询前按模块、对象及字段权限过滤，敏感内容脱敏；
- 日志查询和导出本身也需权限，并记录审计；
- 不记录密码、Token、Cookie、密钥、完整银行账号、完整联系方式、完整地址、SQL、堆栈、服务器路径或未脱敏导入原文；
- 客户端来源仅用于追溯，不改变用户身份或权限结论。

## 9. 系统设置

### 9.1 已批准入口

系统设置只覆盖 Approved Phase 4 已有入口：用户个人设置、已批准的系统参数、编号规则、提醒设置和系统信息只读展示。Excel 导入、导入任务及操作日志分别复用第 5、8 节统一能力。

### 9.2 用户个人设置

用户个人设置只允许查看当前系统身份及维护 Frozen `users` 已批准的本人可编辑资料。用户不得在个人设置中修改用户名、账号状态、角色、权限、仓库/店铺范围、审计信息或安全锁定字段，不得查看密码哈希。修改成功后重新读取正式用户信息；会话身份、权限和数据范围仍由服务端实时判断。

### 9.3 设置边界

- Frozen 业务规则不能包装为可编辑配置；
- 设置页不得修改数据库状态、API 状态、单据状态机或业务流程；
- 只允许维护 Frozen `system_settings` 已能表达且已批准的通用配置；
- 不新增系统配置表、配置对象、配置状态或敏感字段；
- 高风险设置修改必须显示前后差异、二次确认、权限复核并审计；
- 密钥、数据库连接、内部环境配置、文件路径、Token 和凭证不得在普通页面展示；
- 系统信息只读展示批准公开的版本和运行摘要，不暴露基础设施细节。

API Master Specification v1.1 未登记独立系统设置管理 API，本文只定义 Approved 页面产品规则，不虚构接口编号或参数。

## 10. 消息与提醒

### 10.1 轻量提醒范围

本系统只提供轻量提醒，不建立独立复杂消息中心。可展示：

- 待办事项；
- `INV-008` 至 `INV-012` 的库存预警；
- 导入完成或失败；
- 导出完成或失败；
- 登录及安全提醒；
- 已批准的业务异常提醒。

### 10.2 提醒规则

- 提醒是既有业务事实、任务结果或安全事件的派生展示，不是业务状态；
- 关闭或标记已查看提醒不得修改来源业务事实；
- 库存预警处理只调用 Frozen `INV-*` 动作；
- 导入/导出提醒从正式任务或审计结果派生；
- 提醒内容在生成和读取时按功能、记录、仓库、店铺、厂家及字段权限过滤；
- 点击提醒只能跳转至用户有权访问的 Approved 正式页面；
- 无权限、对象失效或范围变化时显示安全提示，不泄露编号、名称、数量或金额；
- 不提供独立消息创建、发送、已读或删除 API，不新增消息表、对象或状态。

## 11. 统一编号能力

- 所有正式业务编号由服务端统一编号能力生成；
- 业务模块不得由前端、本地缓存或各自实现生成编号；
- 编号规则只能使用 Approved `system_settings` 现有配置和 Frozen 唯一约束；
- 业务创建接口不接受客户端伪造的正式编号；
- 编号生成失败时，创建事务整体失败，不得形成无编号正式业务数据；
- 并发生成必须保持正式唯一范围，不得用重试形成重复单据；
- 编号规则入口只展示/维护已批准规则，不重新定义 Frozen 编号字段或唯一范围；
- 无独立 Frozen 编号 API，编号通过各业务创建 API 的服务端行为形成，本文不新增接口。

## 12. 统一数据字典与基础资料

- 状态、枚举和动作名称来自 Frozen 状态机与 API，不由页面硬编码可变副本；
- 仓库、平台、店铺、厂家、产品、SKU、供应商等使用 `MD-*` 正式列表、详情和授权选择接口；
- 低风险字典可按 Task 6.1 短期缓存，状态、库存和权限不得依赖过期缓存；
- 业务模块不得维护重复字典、基础资料或平行编号映射；
- 停用资料不得进入新业务选择，历史单据继续显示正式快照；
- 字典加载失败必须标明失败区域，不得回退到未授权缓存；
- Approved 业务字典入口不得改变 Frozen 枚举、状态机或 API；
- 不新增数据字典表、字段或 API。

## 16. 统一查询、状态动作、异常与删除原则

### 16.1 查询

列表统一支持现有查询、筛选、白名单排序、分页、查看和适用导出。查询变化返回第一页；无数据、筛选无结果、无权限和加载失败必须区分。所有查询在聚合、排序和分页前应用数据范围与字段权限。

### 16.2 状态动作

保存、提交、审核、驳回、撤回、反审核、确认、取消、作废、执行等名称和语义不得随意替换。状态动作只调用 Task 6.1 和 Frozen API 的专用动作，客户端不得直接提交目标状态或乐观修改正式状态。

### 16.3 异常提示

成功、失败、权限不足、记录不存在、数据冲突、校验错误、限流、网络异常和服务不可用均使用 Task 6.1 统一反馈。重要错误显示安全原因、可执行下一步及 Request ID，不暴露数据库、Token、路径或堆栈。

### 16.4 删除与历史保护

- 正式业务数据原则上不得物理删除；
- 基础资料使用启用/停用；
- 正式业务使用取消、作废、撤销或正式冲销；
- 附件按 Frozen 规则解除关联、失效或在极有限条件下删除；
- 日志和正式状态/审批历史永久保留，不提供普通删除；
- 前端不得通过隐藏请求、直接数据库操作或非正式接口绕过历史保护。

## 17. 公共能力统一 16 章模板

本节按 Task 6.1 强制模板逐章覆盖第 3 至 16 节全部公共能力；各能力的具体规则和 Frozen 依据以上文为准。

### 17.1 功能目的

以统一身份、权限、任务、文件、日志、查询、编号、字典、打印、文本和时间规则支撑九个已批准业务模块，避免重复实现和平行事实来源。

### 17.2 使用角色

全体内部用户使用登录、会话、查询、字典、时间和授权范围内的提醒；系统管理员或获授权管理员使用用户、角色、权限和设置；业务用户按权限使用导入、附件、导出和打印；审计人员使用日志查询与导出。

### 17.3 前置条件

除登录入口外，用户必须已认证且账号启用；每项能力必须具备功能、操作、记录和字段权限，目标对象存在且状态合法；微信首次授权须绑定已有启用账号；高风险操作须满足职责和确认条件。

### 17.4 页面入口

只使用 Approved Phase 4 的登录页、微信“我的”、用户权限、系统设置、业务详情附件区、导入/导出入口、日志、Dashboard 待办/预警和已有打印入口，不新增页面或一级模块。

### 17.5 初始化与数据加载

按身份、页面权限、能力摘要、字典选项、主数据和辅助数据顺序加载；只加载当前页必要数据。权限、状态、日志、库存和任务结果不得依赖过期缓存。

### 17.6 字段与显示规则

只显示 Frozen 字段、正式派生值和 Approved 文案；密码、Token、密钥、存储路径和完整敏感值永不显示；金额、成本、个人信息、日志、附件及导入原文按字段权限隐藏或脱敏。

### 17.7 按钮与操作

按钮按身份、权限、状态、范围和风险显示、禁用或隐藏；登录、刷新、退出、用户/权限维护、导入执行、附件删除、敏感导出和设置修改均使用明确动作及高风险确认。

### 17.8 表单校验

统一校验必填、格式、长度、唯一、引用、文件、模板、行级业务、状态、权限、数据范围、版本和幂等；前端校验不替代服务端验证。

### 17.9 API 调用

登录会话只映射 `SEC-*`，导入只映射 `IMP-*`，附件只映射 `ATT-*`，日志只映射 `LOG-*`；导出复用各业务 Frozen 导出接口。无独立 API 的用户/角色/权限维护、设置、提醒、编号、字典、打印、国际化和系统时间明确为 Approved 产品规则或复用现有业务 API，不虚构编号或契约。

### 17.10 状态变化

公共能力不得自行增加状态。会话、导入任务、附件、用户启停、角色启停、库存预警及业务状态只使用 Frozen 语义；提醒、国际化、打印和系统时间不形成业务状态。

### 17.11 权限控制

所有读取、写入、下载、导出、打印和跳转均在服务端重新校验身份、功能、操作、角色、记录、仓库、店铺、厂家派生范围和字段权限。客户端来源、隐藏按钮或缓存摘要不作为授权依据。

### 17.12 成功反馈

成功反馈与服务端正式结果一致；登录进入授权首页，保存刷新最新对象，导入/导出显示任务结果，附件显示正式关联，权限修改显示影响范围。结果未知时不得宣告成功。

### 17.13 异常处理

按 Task 6.1 统一处理 400、401、403、404、409、422、429、500 和 503；保留安全输入和 Request ID，结束 Loading，避免重复写入，不暴露内部信息。

### 17.14 边界情况

覆盖账号停用、会话失效、权限变化、微信绑定账号失效、重复文件、部分成功、失败行重试、附件证据保护、下载权限变化、导出过期、日志脱敏、设置越界、提醒来源失效、编号并发和客户端时间篡改。

### 17.15 日志与审计

登录、用户/角色/权限变更、导入、导出、附件、设置、高风险打印、权限拒绝和安全事件统一写入可映射的 `audit_logs`，只追加、脱敏并关联 Request ID；未知身份事件不得伪造对象。

### 17.16 验收标准

逐项验证双端统一身份、首次微信绑定、停用阻断、权限实时生效、导入校验执行分离、附件重新鉴权、导出范围和脱敏、六类日志统一来源、设置边界、提醒不改事实、编号唯一、字典单一来源、打印授权、中文默认及可信时间；不得新增数据库/API/页面/模块/状态或技术实现。
